# ADR-005: Simplified Event Handling for Stylus and Touch in PhialeGis Drawing Components

## Status
Proposed 02-02-2024

## Context
In the development of PhialeGis components for WPF and UWP, a significant challenge has been the inability to distinguish between Stylus and Touch events effectively, as they are triggered sequentially. This has led to complexities in handling user input, especially in scenarios where precision and responsiveness are paramount. Given the diverse range of devices and the varying capabilities of stylus and touch inputs, it's crucial to streamline the event handling process to ensure a consistent and efficient user experience across both WPF and UWP platforms.

## Decision
To address these challenges, the following approach will be adopted:
1. **Treating Touch as a Special Case of Stylus**: Simplify the event handling by considering Touch events as a special category of Stylus events. This will reduce the complexity of having to handle these events separately and improve the application's responsiveness.
2. **Blocking Touch Events Programmatically**: To prevent redundancy and potential conflicts between Stylus and Touch events, Touch events will be programmatically blocked in WPF by setting `Handled = true` for Stylus events. This ensures that once a Stylus event is recognized, it is prioritized, and the corresponding Touch event is disregarded.
3. **Optional Mouse Event Triggering**: Introduce an option to control whether Mouse events should be triggered when interacting with the component via Stylus or Touch. If this option is set to false, Mouse events will be ignored during Stylus or Touch interactions, which can be achieved by checking for a null `TouchDevice` in Mouse event handlers and returning early. This allows for more focused handling of input events and prevents unintended interactions.
4. **Adherence to Supported System Gestures**: The interpretation of events will follow the guidelines and behaviors outlined in the Microsoft documentation for Supported System Gestures, ensuring compatibility and leveraging built-in gesture recognition where possible. https://learn.microsoft.com/en-us/windows/win32/tablet/system-events-and-mouse-messages#supported-system-gestures

## Consequences
- **Positive**:
  - Simplifies the event handling logic, making it easier to maintain and extend.
  - Improves responsiveness and user experience by reducing the potential for conflicting input events.
  - Provides flexibility in handling Mouse events, allowing developers to tailor the behavior to specific application needs.
- **Negative**:
  - May limit the ability to differentiate between Stylus and Touch inputs in scenarios where such distinction is beneficial.
  - Requires careful implementation to ensure that the blocking of Touch events does not inadvertently affect other functionalities.

## Alternatives Considered
- **Seeking Different Hardware or Configuration**: This approach was rejected as it would tie the implementation too closely to specific hardware setups, limiting the application's versatility.
- **Distinguishing Between Stylus and Touch Based on Specific Features**: Considered but ultimately rejected for now due to the added complexity and potential delays in development.
- **Creating an Abstract Device Concept**: Rejected as an unnecessary complication that would not effectively address the underlying issue.

## Rationale
The chosen approach prioritizes speed and efficiency in development, with a willingness to iterate and make corrections as needed rather than being paralyzed by indecision in the face of many unknowns. This strategy aligns with the project's goals of delivering a robust, user-friendly GIS application that performs well across a wide range of devices.
