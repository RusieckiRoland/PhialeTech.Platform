# ADR-014: YAML UI Enterprise Component Palette Architecture

## Status

Accepted

## Context

The repository needs a reusable enterprise component palette for the future YAML UI runtime.
This stage does not implement a YAML parser, inheritance, or merge engine.
It establishes the product architecture, contracts, and WPF-first implementation path without breaking portability to WinUI 3 and Avalonia.

The palette must follow these constraints:

- platform-neutral contracts
- a strict Core layer for component semantics
- Universal Inputs as the only allowed interaction bridge
- thin platform fronts
- no direct front-to-core calls
- one shared style library
- validation as part of the component contract

## Decision

The YAML UI palette is introduced as a new product under:


The interaction rule is explicit:

- `front -> Universal Inputs -> Core`
- no direct front-to-core communication

This means:

- `Contracts` define component definitions, state contracts, validation metadata, and neutral rendering data
- `Core` owns component semantics, defaults, validation, visibility rules, and input processing
- `WPF` renders neutral state and maps WPF events into `UniversalInput.Contracts`
- shared visuals stay in `src/PhialeTech/Shared/PhialeTech.Styles.Wpf`

## Consequences

Positive consequences:

- the palette stays aligned with the portability goal
- future WinUI 3 and Avalonia fronts can reuse the same contracts and core logic
- demo integration can happen without embedding business logic into controls
- future YAML `extends`, `id` overrides, and deterministic merge can target stable contracts

Tradeoffs:

- composition must happen through neutral state and input contracts, which adds structure up front
- new controls cannot shortcut logic into WPF code-behind

## Implementation Notes

WPF is implemented first, but the project layout already reserves the same product split that later platform fronts will follow.

The initial WPF milestone covers:

- architecture skeleton
- contracts and validation models
- a thin renderer and component host
- showcase integration in `PhialeTech Demo`

## Guardrails

- no platform-specific styling inside component projects
- no hidden defaults depending on visual tree position
- no contract design that depends on YAML parser behavior
