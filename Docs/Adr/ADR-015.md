# ADR-015: Shared Application Shell and Hosted Surface Architecture

* **Status:** Accepted
* **Date:** 2026-04-18
* **Decision Maker:** Roland Rusiecki
* **Related:** `Docs/Adr/ADR-009.md`, `Docs/Adr/ADR-011.md`, `Docs/Adr/ADR-012.md`, `Docs/Adr/ADR-014.md`

## Context

The repository now contains multiple reusable products and showcase applications that need a consistent desktop shell and a consistent way to present hosted content such as dialogs, sheets, modal flows, and view-backed overlays.

Before this step, the codebase had two growing problems:

1. shell behavior and page framing were starting to accumulate inside host-specific windows,
2. transient hosted UI flows were at risk of being implemented ad hoc inside WPF code-behind and product-specific surfaces.

This would create architectural drift:

* shell layout would become a property of the first host instead of a reusable product,
* modal and overlay behavior would be reimplemented separately by each frontend,
* products such as YAML UI would start depending on WPF windowing details instead of neutral presentation contracts,
* future Avalonia and WinUI 3 ports would inherit WPF-specific assumptions.

The project needs one explicit decision for both concerns:

* the application shell must be modeled as a reusable shared product with neutral presentation state,
* hosted surfaces must be modeled as a reusable presentation capability with neutral contracts and platform-specific rendering adapters.

## Decision

We introduce two shared architectural products under `src/PhialeTech/Shared`:

1. `PhialeTech.Shell.*`
2. `PhialeTech.ComponentHost.*`

They are related but distinct.

### 1) Shell is a reusable product, not a host-local window convention

`PhialeTech.Shell.Abstractions` and `PhialeTech.Shell` define the neutral shell presentation model.

This model owns:

* navigation items
* title and subtitle semantics
* command collections
* status collections
* validation rules for shell state consistency

Frontend implementations such as WPF render this model through thin shell controls (`PhialeWindow`, `PhialeTitleBar`, `PhialeAppShell`) but do not redefine the shell contract.

### 2) Hosted surfaces are a reusable presentation capability

`PhialeTech.ComponentHost.Abstractions` and `PhialeTech.ComponentHost` define the neutral model for hosted surfaces.

This model owns:

* hosted surface request
* hosted surface kind
* hosted presentation mode and size
* session state
* result contract
* shell coordination boundary

Frontend implementations such as WPF provide:

* modal layer rendering
* visual chrome
* host-specific view factories
* Universal Input bridging from platform events into hosted content flows where needed

### 3) Product features request hosting through contracts, not through window code

Feature products such as YAML UI must not open dialogs or sheets by directly orchestrating WPF windows or overlay trees.

Instead, they must communicate through hosted surface contracts and coordinators.

The required direction is:

* `feature/demo/frontend -> hosted surface manager/coordinator -> platform host renderer`

not:

* `feature/demo/frontend -> ad hoc window/dialog code`

### 4) Shell and hosted surfaces stay separate

The shell provides the stable application frame.
The component host provides transient presentation layers inside or above that frame.

They may collaborate, but neither should absorb the other’s responsibilities.

Specifically:

* shell does not become a modal manager,
* component host does not become the global page shell,
* product runtimes do not depend on either through platform-specific types.

## Architectural Constraints

1. Public contracts for shell and hosted surfaces live in shared abstractions only.
2. Core logic for shell validation and hosted surface orchestration lives in shared non-platform projects.
3. WPF-specific primitives such as routed events, dependency properties, visual tree assumptions, and `Window` ownership must not leak into abstractions or Core.
4. Frontends may render and bridge, but may not own presentation semantics that belong to the neutral model.
5. Hosted surfaces must remain compatible with future Avalonia and WinUI 3 implementations.
6. No silent fallback path is allowed where missing shell or hosted-surface integration is hidden behind local window hacks.

## Consequences

### Positive

1. The repository gains one reusable shell model instead of repeated application framing logic.
2. Hosted dialogs, sheets, and overlays can be introduced consistently across products.
3. YAML UI and future products can request hosted presentation without coupling themselves to WPF internals.
4. Avalonia and WinUI 3 ports have a clear architectural target: implement rendering adapters, not invent a second model.
5. Validation of shell state becomes explicit and testable.

### Costs / Risks

1. The number of shared projects increases, which raises structural complexity.
2. Features that previously could have been implemented directly in WPF now require neutral contracts and coordinators.
3. Host integration takes longer up front because the architecture forbids local convenience shortcuts.
4. Poor boundary discipline could cause overlap between shell state, hosted presentation state, and feature state if not reviewed carefully.

## Implementation Notes

The first production-grade host is WPF, but the architecture is not WPF-owned.

The initial implementation includes:

* shell abstractions and state validation
* WPF shell controls
* component host abstractions and manager
* WPF modal layer host and hosted-surface service
* demo integration through host factories and coordinators

Future hosts must implement equivalent rendering and integration behavior while preserving the neutral contract shape.

## Guardrails

* No product may directly encode modal presentation behavior into WPF window code if the same behavior belongs in hosted surface contracts.
* No product-specific shell may redefine command/status/navigation semantics outside the shared shell model unless a new shared abstraction is explicitly approved.
* No platform-specific type may appear in shared shell or hosted-surface contracts.
* Do not merge shell state and hosted-surface state into one catch-all presentation object.
* Prefer explicit failure when host infrastructure is missing; do not add hidden rescue paths.
