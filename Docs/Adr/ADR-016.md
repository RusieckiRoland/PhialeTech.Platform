# ADR-016: YAML Form Composition Model with Action Shells and Structured Regions

* **Status:** Accepted
* **Date:** 2026-04-18
* **Decision Maker:** Roland Rusiecki
* **Related:** `Docs/Adr/ADR-011.md`, `Docs/Adr/ADR-014.md`

## Context

The initial YAML UI ADR established the enterprise component palette architecture and the rule that interaction must flow through Universal Inputs into Core.

That decision was intentionally broad.

The next stage required a more specific answer:

* what is the runtime model of a YAML form,
* how reusable form shells should be expressed,
* where header, footer, and action regions belong,
* how generated forms should be hosted and rendered without creating a second, unrelated document model.

Without an explicit decision, the implementation would drift into one of these failure modes:

1. forms become a special WPF-only concept detached from the shared document model,
2. reusable headers and footers are implemented as frontend templates instead of definition-level composition,
3. actions are attached as local button strips rather than as part of document semantics,
4. generated forms and authored forms diverge into separate runtime paths.

The project needs one explicit model so that composition, rendering, and future YAML merge behavior stay aligned.

## Decision

We adopt a **single structured document model for forms**, where forms are specialized YAML documents composed from reusable shells and region definitions.

### 1) A form is a structured document, not a separate host-only control family

Forms are modeled inside the YAML document runtime rather than as a disconnected frontend-only feature.

This means:

* form semantics live in shared contracts and Core,
* the renderer consumes resolved document/form state,
* platform fronts render the resolved structure without owning form meaning.

### 2) Reusable form shells are first-class composition inputs

Reusable action shells and form shells are part of the definition system.

They may provide:

* header structure
* footer structure
* action area placement
* region chrome semantics
* standard shell composition for enterprise scenarios

These shells are not frontend templates.
They are definition-layer composition units intended for authored and generated forms alike.

### 3) Header, footer, and action regions belong to the document model

Header and footer regions, including their actions and chrome semantics, are part of the resolved document definition.

They are not a post-processing decoration step in WPF.

This means:

* action placement is part of the normalized and resolved definition,
* render-plan building happens from Core-owned document semantics,
* WPF hosts render those regions but do not invent them.

### 4) Generated forms reuse the same model as authored forms

Generated form flows must compile into the same resolved runtime model used by authored YAML forms.

The system must not create:

* one model for authored forms,
* another model for generated or demo-driven forms.

Generated forms may differ by source, but not by architectural contract.

### 5) Visual primitives remain components inside the same model

Buttons, badges, containers, and related visual primitives are part of the YAML UI component palette and participate in the same document/form composition model.

Their semantics belong to shared contracts and Core.
Their WPF/Skia presentation belongs to platform rendering.

## Architectural Constraints

1. Composition semantics must stay in shared contracts and Core, not in WPF markup conventions.
2. Reusable action shells must be representable at the definition level.
3. Header/footer/action region behavior must not depend on visual tree position or host-specific template assumptions.
4. Generated forms must compile through the same normalized and resolved pipeline as authored forms.
5. New form features must extend the structured document model instead of creating parallel models.
6. Interaction from rendered form controls still follows the rule from ADR-014: `front -> Universal Inputs -> Core`.

## Consequences

### Positive

1. One runtime model can support authored forms, generated forms, inherited shells, and reusable action layouts.
2. Header/footer/action semantics become portable across WPF, WinUI 3, and Avalonia.
3. Demo scenarios and future production forms exercise the same composition pipeline.
4. Buttons, badges, and future enterprise primitives can evolve without breaking the document architecture.
5. Future `extends`, override, and merge behavior has a stable structural target.

### Costs / Risks

1. The shared model becomes richer and therefore more expensive to evolve carelessly.
2. Normalization and resolution logic must stay disciplined; otherwise the pipeline becomes hard to reason about.
3. Frontend developers lose the option of solving layout edge cases with local WPF-only shortcuts.
4. Backward-compatibility shims would be especially harmful here because they would split the composition model.

## Implementation Notes

The current implementation direction includes:

* form document specialization in shared contracts
* normalized/resolved action areas
* structured header and footer definitions
* reusable action shell YAML definitions
* render-plan building from Core-owned semantics
* WPF rendering through `YamlDocumentHost` and related controls
* generated form demo flows reusing the same document runtime

Future merge and inheritance work must extend this model rather than replace it.

## Guardrails

* Do not implement reusable form shells as WPF-only templates.
* Do not introduce a second runtime path for generated forms.
* Do not move action/header/footer semantics into platform rendering code.
* Do not add local fallback behavior that hides missing definition data.
* Prefer explicit resolution errors over silent defaults when composition data is incomplete.
