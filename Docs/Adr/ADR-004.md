# ADR-004: Event Handling in PhialeGis Drawing Components for WPF and UWP Platforms

## Status
Applied 24-01-2024

## Context
The PhialeGis project, aimed at GIS applications, requires advanced event handling for drawing components in both WPF and UWP versions. Considering the advanced touch capabilities of UWP, especially the distinction between pen, touch, and mouse inputs, it's strategic to align the event handling of WPF components with that of UWP. This approach caters to the GIS application users who need various tools for map viewing and editing. The market already offers similar products, so prioritizing user convenience is essential. To facilitate this, a universal event handling library is introduced, streamlined for rapid implementation yet open for future expansion.

## Decision
To harmonize the event handling in WPF and UWP versions, the following approach is adopted:
1. **Event Handling Alignment with UWP**: The event handling in WPF will be adapted to match the UWP's advanced touch capabilities.
2. **Universal Event Handling Library**: A streamlined library is introduced to ensure common functionality across platforms, focusing on implementation speed and future scalability.
3. **Variation of User Actions**: User actions will be categorized into mouse, touch, and pen inputs, with a structural Bridge pattern proposed to avoid excessive implementation expansion for each input type.
4. **Finite State Machine (FSM) for Command Handling**: To manage user commands effectively, an FSM with a Transition Table approach will be used. The basic class for command handling, 'BaseActivity', has been preliminarily implemented.

## Consequences
- **Positive**:
  - Enhances user experience by providing advanced touch capabilities in GIS applications.
  - Ensures a consistent approach to event handling across WPF and UWP platforms.
  - Streamlines the development process with a universal event handling library.
- **Negative**:
  - Introduces complexity in aligning event handling capabilities of WPF with UWP.
  - Requires careful validation of the Bridge pattern's effectiveness in managing diverse input methods.
  - Balancing rapid implementation with user convenience may lead to initial feature limitations in WPF.

## Alternatives Considered
- Keeping separate event handling mechanisms for WPF and UWP was considered but rejected due to inconsistency in user experience.
- A more comprehensive library for event handling was evaluated but deemed too time-consuming for the current implementation phase.

## Future Consideration
The effectiveness of the Bridge pattern in handling different input methods will be reviewed in practice. The possibility of expanding the event handling library in future development spirals will be open, allowing for enhancements based on user feedback and technological advancements.

## Notes
This ADR focuses on balancing implementation speed with the strategic goal of user convenience in a competitive GIS application market. The decision leans towards creating a robust and scalable solution that can adapt to evolving requirements and user expectations.
