# ADR-017: Explicit Overlay Scopes for Hosted Controls

* **Status:** Accepted
* **Date:** 2026-04-28
* **Decision Maker:** Roland Rusiecki
* **Related:** `Docs/Adr/ADR-015.md`, `Docs/Adr/ADR-016.md`

## Context

The application contains reusable hosted controls that can be expanded or overlaid, including PDF viewer surfaces and rich text/document editor surfaces.

The first implementations evolved differently:

* one control presented a lightweight floating icon and a local focus mode,
* another control used a more complete overlay mechanism that could expand the editor surface,
* the behavior was not tied to one explicit rule about where an overlay is allowed to expand.

This creates a UX and architecture problem.

Users reasonably expect the same overlay affordance to behave the same way across controls. At the same time, the correct overlay boundary is not always the whole application window.

Examples:

* a PDF viewer placed directly on a demo page may expand to the page work area,
* a text editor placed inside a modal form should expand only inside that modal form,
* a text editor placed inside a panel or card may need to expand only inside that panel,
* a dynamic YAML form must express the same rule without WPF-specific markup.

The phrase "nearest sensible container" is not acceptable as an implementation rule. In configurable forms, the runtime cannot guess which container is semantically responsible for an overlay. The boundary must be explicit.

## Decision

We adopt **explicit Overlay Scopes** as the only supported rule for determining where hosted controls expand.

An overlay-capable control does not decide by itself whether it expands to a card, modal, page, or window. It requests overlay presentation from its host. The host resolves the request against the nearest explicitly declared overlay scope.

The rule is:

**A control overlay expands to the nearest ancestor marked as an overlay scope.**

If no overlay scope exists, a nearest-scope overlay request fails. The host must not silently promote the request to the page, window, or application root.

### 1) Overlay scope is an explicit semantic marker

An overlay scope is a container-level declaration that says:

* controls inside this container may expand within this container,
* the container is responsible for clipping, focus, sizing, and restoration boundaries,
* the overlay must not escape this boundary unless the control or host explicitly requests a higher scope.

Overlay scope is not inferred from visual appearance.

The runtime must not guess based on:

* border radius,
* card chrome,
* modal-looking layout,
* control size,
* position in the visual tree alone,
* WPF element type such as `Grid`, `Border`, or `Window`.

### 2) Static WPF forms declare overlay scope with an attached property or wrapper

For regular, non-dynamic WPF forms, the form author marks the intended overlay boundary in XAML or code.

Preferred XAML shape:

```xml
<Grid phiale:OverlayHost.IsScope="True">
    <Grid>
        <!-- form body -->
        <phiale:DocumentEditor />
    </Grid>
</Grid>
```

Equivalent C# shape:

```csharp
OverlayHost.SetIsScope(reviewDialogBody, true);
```

The WPF implementation uses `OverlayHost.IsScope` as the attached property. The target element must be a `Panel`, because the overlay presenter is layered into that panel.

Valid static-form scopes include:

* modal body containers,
* modal content roots,
* panel bodies,
* editor sections,
* page work areas.

Static WPF implementation may use visual-tree lookup to find the nearest scope, but only scopes explicitly marked with the attached property are valid.

### 3) Dynamic YAML forms declare overlay scope in the definition

For dynamic/configurable forms, overlay scope must be part of the resolved definition model.

Example:

```yaml
layout:
  type: Column
  items:
    - type: Container
      caption: Reviewer
      items:
        - type: Row
          items:
            - fieldRef: firstName
            - fieldRef: lastName
    - type: Container
      caption: Review notes
      overlayScope: true
      items:
        - fieldRef: notes
```

The dynamic form compiler must carry this information into the resolved render model. Platform renderers then mark the corresponding platform container as an overlay scope.

YAML overlay semantics belong to the definition/runtime model, not to WPF-only rendering shortcuts.

### 4) Overlay-capable controls request overlay; hosts resolve it

Controls such as `PdfViewer`, `DocumentEditor`, `MonacoEditor`, or future WebHost-backed controls should not each invent their own unrelated overlay behavior.

The preferred interaction is:

1. the control exposes a lightweight overlay affordance or sends an overlay request,
2. the host receives the request,
3. the host resolves the nearest overlay scope,
4. the host presents the control within that scope,
5. Escape, focus restoration, and close behavior are handled consistently.

The visual affordance may be control-specific only where the control surface requires it. The baseline visual style should be shared:

* small floating button,
* semi-transparent background,
* stable icon,
* consistent tooltip,
* consistent expand/collapse state.

### 5) What happens when no overlay scope exists

No-scope behavior must be explicit. It must not depend on accidental visual-tree structure.

The default policy is fail-fast for nearest-scope overlays:

1. If the overlay-capable control is configured for scoped overlay and no `OverlayHost.IsScope` ancestor exists, the request fails with a visible configuration error.
2. If overlay should be unavailable in a scope-less form, the control must be explicitly configured as disabled.
3. If full-window overlay is desired, that must be an explicit overlay mode, not a fallback from missing scope.

For dynamic YAML forms this means that `overlayScope: true` belongs on the layout node that is meant to bound the overlay. `layout`, `row`, `column`, and `container` nodes may be scopes. Leaf items such as `fieldRef`, badges, and buttons are not valid scope providers.

## Architectural Constraints

1. Overlay boundaries must be explicit.
2. Static WPF forms use an attached property or wrapper to mark overlay scopes.
3. Dynamic forms use definition-level scope declarations that survive normalization and render-plan building.
4. Overlay-capable controls request overlay presentation; they do not own scope resolution.
5. Scope resolution belongs to host infrastructure, not to individual component demos.
6. Platform-specific mechanisms such as WPF visual-tree walking must not leak into shared YAML or component contracts.
7. Missing overlay scope for nearest-scope overlays is a configuration error; silent arbitrary fallback is forbidden.

## Consequences

### Positive

1. Overlay behavior becomes predictable across PDF viewer, document editor, and future hosted controls.
2. Modal forms can safely contain rich editors that expand only to modal boundaries.
3. Dynamic YAML forms can express overlay behavior without relying on WPF element structure.
4. The UI can reuse a consistent floating overlay affordance while preserving context-specific expansion.
5. Future hosts such as WinUI 3 and Avalonia have a clear contract to implement.

### Costs / Risks

1. Form authors must explicitly mark overlay scopes when they want bounded expansion.
2. Dynamic form schema and render-plan models need additional overlay metadata.
3. Existing ad hoc overlay behavior must be migrated or wrapped.
4. Scope fallback policy must be tested carefully to avoid surprising full-window expansion.
5. There is a short-term implementation cost to centralize behavior that currently exists in multiple controls.

## Implementation Notes

The likely WPF implementation includes:

* `OverlayHost.IsScope` attached property,
* `OverlayHost.FindNearestScopePanel(DependencyObject source)` helper,
* shared floating overlay button style,
* host-level overlay presenter that can reparent or visually layer a hosted control within the resolved scope,
* Escape handling and focus restoration owned by the host,
* diagnostics when scope resolution fails.

For YAML forms, implementation should add:

* `overlayScope` to `layout`, `row`, `column`, and `container` definitions,
* resolved render metadata so platform renderers can mark scope containers,
* validation that rejects `overlayScope` on leaf layout items.

Existing `DocumentEditor` overlay behavior is closer to the desired expansion model, but its scope resolution should become host-owned and reusable.

Existing `PdfViewer` floating icon is closer to the desired affordance style, but its expansion behavior should be migrated away from showcase-specific layout changes.

## Guardrails

* Do not infer overlay scopes from "modal-looking" or "card-looking" visuals.
* Do not let each component implement a different expand/focus behavior.
* Do not use full-window overlay as an invisible default for controls inside modal forms.
* Do not encode dynamic form overlay semantics only in WPF templates.
* Do not allow scope fallback to hide malformed dynamic form definitions.
* Prefer explicit author intent over convenience heuristics.
