# ADR: Naming Rules for Domain Models, DTOs, Contracts, and Product Assemblies

* **Status:** Accepted
* **Date:** 2026-03-10
* **Owners:** GIS Platform Team
* **Related:** `PhialeGis.Library.*`, `PhialeTech.*`, `UniversalInput.Contracts`, `PhialeTech.ComponentHost.Abstractions`, future styling and picker systems

---

## Context

The repository is now evolving along two connected but distinct architectural paths:

1. **GIS infrastructure and rendering**, which belongs to `PhialeGis.Library.*`
2. **Reusable UI products**, which belong to `PhialeTech.*`

At the same time, the codebase is starting to accumulate several kinds of types:

* domain models
* catalog definitions
* UI-facing state models
* host abstraction contracts
* serialized payloads
* import/export formats

Without explicit naming rules, these concerns become mixed. The most common failure modes are:

* using `Dto` for domain models that are actually part of the core semantic model,
* leaking transport naming into shared runtime APIs,
* using product naming for GIS infrastructure,
* exposing application-specific semantics from reusable component contracts,
* making it unclear which types are stable public boundaries and which are internal runtime structures.

We therefore need a repository-wide naming decision that reflects architectural boundaries rather than cosmetic style preferences.

---

## Decision

### 1. `Dto` is reserved for transport, persistence, and boundary payloads

The suffix `Dto` must be used only for types whose primary purpose is one of the following:

* file serialization
* import/export payloads
* inter-process or inter-service transport
* external integration contracts where a payload shape is the point of the type

Examples:

* `StyleCatalogDto`
* `ImportedSymbolDto`
* `LayerSnapshotDto`

`Dto` must **not** be used for the main semantic model of the runtime if that model is part of the architecture itself.

### 2. Shared semantic models use domain-oriented names

If a type represents the real model used by the shared logic, it must use a domain name rather than `Dto`.

Preferred patterns:

* `*Definition`
* `*Descriptor`
* `*Request`
* `*Result`
* `*Options`
* `*State`
* `*Snapshot`

Examples:

* `SymbolDefinition`
* `LineTypeDefinition`
* `FillStyleDefinition`
* `GridSummaryDescriptor`
* `StylePreviewRequest`

### 3. Contracts are named by responsibility, not by transport shape

Interfaces must communicate responsibility clearly.

Preferred patterns:

* `I...Catalog`
* `I...Resolver`
* `I...Renderer`
* `I...Service`
* `I...Provider`
* `I...Policy`
* `I...Host`
* `I...State`

Examples:

* `ISymbolCatalog`
* `IStylePreviewService`
* `IPhialeLayerCollectionState`
* `IStyleContrastPolicy`

### 4. GIS infrastructure stays under `PhialeGis.Library.*`

If a module belongs to GIS runtime behavior, rendering infrastructure, geometry systems, style resolution, or application rendering semantics, it must remain under the `PhialeGis.Library.*` naming path.

Examples:

* `PhialeGis.Library.Abstractions`
* `PhialeGis.Library.Core`
* `PhialeGis.Library.Renderer.Skia`

This includes:

* style engines
* render pipelines
* preview rendering services used by the GIS runtime
* domain-level style definitions

### 5. Reusable product components use the `PhialeTech.*` product path

If a module is a reusable product component or a platform host for such a component, it must use the `PhialeTech.*` naming path.

Examples:

* `PhialeTech.Grid.Core`
* `PhialeTech.Grid.Wpf`
* `PhialeTech.ActiveLayerSelector`
* future `PhialeTech.LineStylePicker`

### 6. Bridge libraries must be explicitly named as abstractions

Libraries that exist only to define the boundary between systems must use names that make that role explicit.

Preferred pattern:

* `*.Abstractions`

Examples:

* `PhialeTech.ComponentHost.Abstractions`
* future `PhialeTech.Gis.Abstractions`

These libraries must contain:

* neutral contracts
* integration-facing state interfaces
* event args
* snapshots or DTOs only when required by the boundary

They must not contain platform rendering logic or application orchestration.

### 7. Platform projects may keep platform-specific root namespaces if needed, but product identity must be explicit

For reusable products:

* project identity,
* package identity,
* visible solution naming

must clearly reflect the product path.

Examples:

* `PhialeTech.Grid.Wpf`
* `PhialeTech.Grid.WinUI`
* `PhialeTech.Grid.Avalonia`

Internal namespaces may preserve compatibility where necessary during migration, but the published identity must follow the product naming rule.

### 8. Preview and runtime rendering must share semantic names

When the same style system is used both for:

* map rendering
* picker preview rendering

the shared model must retain one semantic naming system.

That means:

* the renderer may expose preview-specific request objects,
* but the style definitions themselves remain shared semantic models,
* and should not be duplicated as separate UI-only `Dto` types.

---

## Rationale

* Prevents confusion between domain models and serialized payloads.
* Makes architectural boundaries visible in code.
* Reduces the chance that reusable components leak GIS-specific runtime details.
* Keeps `PhialeTech` product APIs cleaner and more extractable.
* Supports future style pickers that reuse the same style engine as `DrawBox`.
* Creates a stable naming rule for future migration work.

---

## Consequences

### Positive

* Type purpose becomes easier to understand from the name alone.
* Public APIs become more coherent.
* Serialization models and runtime models are less likely to be conflated.
* Future extraction of `PhialeTech` components becomes safer.
* Shared rendering systems can be reused by both GIS runtime and UI preview hosts without duplicate model layers.

### Trade-offs

* Some existing projects and types may keep legacy namespaces temporarily during migration.
* Refactoring away from older names may need staged changes.
* Teams must distinguish carefully between "semantic model" and "transport payload" when creating new types.

---

## Repository Rule

When introducing a new type, the author must first classify it:

1. Is it a runtime semantic model?
2. Is it a transport/persistence payload?
3. Is it a boundary abstraction?
4. Is it a reusable product module?
5. Is it GIS infrastructure?

The name must follow that classification.

---

## Enforcement Rule

Any new proposal must satisfy the following checks:

1. If the type is part of shared runtime semantics, do not use `Dto`.
2. If the type crosses a serialization or integration boundary, `Dto` may be used.
3. If the module is GIS infrastructure, keep it under `PhialeGis.Library.*`.
4. If the module is a reusable product component, use `PhialeTech.*`.
5. If the module is only a cross-system contract boundary, use an `*.Abstractions` library.

If these checks are not satisfied, the proposal conflicts with this decision.

---

## Initial Application of This Rule

This decision applies immediately to the planned style system and its future pickers:

1. Shared style engine types should prefer names such as:
   * `SymbolDefinition`
   * `LineTypeDefinition`
   * `FillStyleDefinition`

2. Serialized style import/export payloads may use:
   * `SymbolDefinitionDto`
   * `LineTypeDefinitionDto`
   * `FillStyleDefinitionDto`
   only if they are truly transport-oriented payloads.

3. GIS style/render logic remains under:
   * `PhialeGis.Library.Abstractions`
   * `PhialeGis.Library.Core`
   * `PhialeGis.Library.Renderer.Skia`

4. Future reusable picker components belong under:
   * `PhialeTech.*`

5. Their host-facing integration contracts belong in:
   * `PhialeTech.ComponentHost.Abstractions`
   or a future dedicated GIS integration abstraction if required.
