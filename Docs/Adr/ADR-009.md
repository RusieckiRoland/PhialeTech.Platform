# ADR: New UI Components Follow the PhialeTech Product Path

* **Status:** Accepted
* **Date:** 2026-03-10
* **Owners:** GIS Platform Team
* **Related:** PhialeGrid, PhialeTech.ActiveLayerSelector, future PhialeTech UI components, DrawBox, DslEditor

---

## Context

The repository now contains two different kinds of UI components:

1. **GIS-native components** that are historically coupled to the PhialeGis application and are part of its core operational model.
2. **Reusable product components** that may later be extracted into other projects, packaged separately, or reused outside the PhialeGis application.

We need a clear architectural rule for all future components so that:

* new components do not become tightly coupled to PhialeGis internals,
* platform-specific hosts stay thin,
* shared logic remains portable across WPF, WinUI, and Avalonia,
* GIS integration is handled through stable abstractions rather than direct application references.

At the same time, we must avoid destabilizing existing GIS foundation modules that are already deeply integrated into the current application.

---

## Decision

### 1. Legacy GIS foundation components remain in the PhialeGis path

The following components are explicitly treated as **GIS-coupled foundation modules** and remain in their current architectural position:

* `DrawBox`
* `DslEditor`

They are not migrated to the `PhialeTech` product structure at this time.

### 2. All new reusable components must follow the PhialeTech product path

Every new component that is intended to be reusable, independently testable, or potentially extractable must be implemented as a `PhialeTech` product.

Each such component must follow this structure:

```text
src/Products/<ComponentName>/
  Core/
    PhialeTech.<ComponentName>/
  Platforms/
    Wpf/
      PhialeTech.<ComponentName>.Wpf/
    WinUI3/
      PhialeTech.<ComponentName>.WinUI/
    Avalonia/
      PhialeTech.<ComponentName>.Avalonia/
```

### 3. Core owns the logic; platform hosts stay thin

For all new `PhialeTech` components:

* `Core` owns:
  * state
  * models
  * view model / presenter logic
  * orchestration
  * command handling
  * localization keys and text resolution
  * platform-neutral interaction logic

* platform projects (`Wpf`, `WinUI`, `Avalonia`) own only:
  * rendering
  * templates and styles
  * platform asset loading
  * platform event wiring
  * thin adapters from platform events to shared interaction contracts

### 4. Interaction between platform hosts and Core must go through UniversalInput

New reusable components must use `UniversalInput.Contracts` as the interaction bridge between platform hosts and portable component logic.

If a required interaction concept is missing from `UniversalInput.Contracts`, the contract library must be extended instead of implementing platform-specific business logic inside the host.

### 5. GIS integration must go through a dedicated abstraction library

Reusable `PhialeTech` components must not directly depend on PhialeGis domain/runtime classes.

A dedicated integration library must be used as the only formal contract between the PhialeGis application and PhialeTech reusable components.

The target integration library is:

* `PhialeTech.Gis.Abstractions`

This library will contain:

* GIS-facing DTOs / snapshots
* GIS event args
* GIS-facing interfaces
* synchronization contracts between application state and reusable components

Example use cases:

* `ActiveLayerSelector` must read layer state through a neutral GIS interface
* `ActiveLayerSelector` must raise state changes through that interface
* `ActiveLayerSelector` must react when the GIS application changes layer state elsewhere

### 6. Public APIs of reusable components must remain application-neutral

Reusable `PhialeTech` components must not expose public APIs that leak PhialeGis-specific implementation details.

They must avoid direct public references to:

* application runtime managers
* GIS-only internal domain classes
* platform-specific UI primitives

Only neutral contracts, DTOs, and abstractions are allowed in reusable component APIs.

---

## Rationale

* Preserves stability of existing GIS foundation modules without forcing a risky migration.
* Ensures that all new components are built for reuse from the start.
* Prevents future coupling between PhialeTech products and PhialeGis internals.
* Makes future WinUI and Avalonia hosts significantly cheaper because logic is already concentrated in `Core`.
* Establishes a single integration boundary for GIS-aware components.

---

## Consequences

### Positive

* New components can be extracted to separate repositories or solutions with much lower effort.
* Platform ports become thinner and more predictable.
* GIS integration becomes explicit and testable.
* Architectural drift is reduced because every new component follows the same structure.

### Trade-offs

* More upfront design work is required for new components.
* `UniversalInput.Contracts` may need to grow over time.
* A new shared integration library (`PhialeTech.Gis.Abstractions`) introduces another contract boundary that must be versioned carefully.

---

## Repository Structure Rule

The target repository structure for new reusable components is:

```text
src/
  Shared/
    UniversalInput.Contracts/
  Integrations/
    PhialeTech.Gis.Abstractions/
  Products/
    PhialeGrid/
    ActiveLayerSelector/
    <NextComponent>/
  PhialeGis/
    ...existing GIS-coupled libraries...

demo/
  Shared/
  Wpf/
  WinUI3/
  Avalonia/

tests/
  ...
```

---

## Explicit Exception Rule

The following modules are exceptions to the new product-path rule and remain in the current application-oriented architecture:

* `DrawBox`
* `DslEditor`

Reason:

* they are strongly coupled to the current GIS application model,
* they are considered part of the GIS foundation,
* migrating them now would create disproportionate risk compared to current value.

---

## Enforcement Rule

Any new UI component proposal must answer these questions before implementation:

1. Is this a reusable product component or a GIS-coupled foundation module?
2. Does the component have a portable `Core`?
3. Is the platform host thin?
4. Does the host communicate with `Core` through `UniversalInput.Contracts` where interaction logic is involved?
5. Does GIS integration go only through `PhialeTech.Gis.Abstractions`?

If the answer is **no** to questions 2-5, the component does not satisfy the architectural rule.

---

## Initial Reference Components

The intended reference direction is:

* `ActiveLayerSelector` -> should follow the `PhialeTech` product path
* `PhialeGrid` -> should continue moving toward the same architecture

These components should act as the pattern for future reusable controls.
