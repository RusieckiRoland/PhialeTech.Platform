# ADR: Global Multi-Viewport Interactive Session, Touch-First Takeover, Central Snapping, and Edit-Action Foundation

* **Status:** Accepted
* **Date:** 2026-03-11
* **Decision Makers:** PhialeGis Architecture / Core Interaction Team
* **Related:** `Docs/Adr/ADR-004.md`, `Docs/Adr/ADR-005.md`, `Docs/Adr/ADR-010.md`, `Docs/Adr/ADR-011.md`

## Context

Current interactive drawing flow (for example `AddLineString`) is session-scoped primarily by `TargetDraw` and preview is stored per viewport target in the interaction manager. This is good for isolated actions, but it does not fully satisfy the required UX and architecture goals:

1. The action geometry should be visible on all viewports that cover the same map extent, not only on the initiating viewport.
2. The active input viewport should be transferable during an ongoing action.
3. Transfer must be touch-first (future monitor/tablet workflows), not keyboard-dependent.
4. Snapping must be introduced consistently across pointer-driven actions.
5. Editing actions should reuse the same interaction pattern used by add actions.

The user explicitly accepted a touch-friendly takeover trigger in a viewport status bar and rejected keyboard-only takeover as the primary mechanism.

## Decision

We adopt a **global action session model per map context** with explicit input ownership and shared preview visibility.

### 1) Session ownership and visibility model

Interactive session state is no longer modeled as "bound to exactly one viewport target."  
Instead, each session has:

1. `SessionId`
2. `MapContextId` (or equivalent map-group identity)
3. `InputViewportId` (single viewport that receives action input)
4. `VisibleInViewportIds` (all viewports where preview/cursor hints are rendered)
5. lifecycle (`Running`, `Suspended`, `Finished`)
6. `CanBeSuspended`

`InputViewportId` can change without recreating the action session.

### 2) Takeover UX (touch-first)

Primary takeover mechanism:

1. Add a viewport status bar outside map canvas.
2. In that bar show `Przejmij rysowanie` button only when:
   - an interactive action is running, and
   - viewport is not current `InputViewportId`.
3. Tapping the button requests transfer of `InputViewportId` to this viewport.

Supporting mechanisms:

1. Optional command-line takeover command (for automation/testing), e.g. `TAKEOVER VIEWPORT <id>`.
2. Keyboard shortcuts may exist as secondary helpers, but are not required for correctness.

### 3) Snapping architecture

Introduce centralized snapping service used by the interaction manager before dispatching pointer input to actions:

1. `ISnapService` in shared abstractions.
2. `SnapRequest` and `SnapResult` typed DTOs.
3. Interaction manager resolves pointer model position and then applies snapping.
4. Action receives already-resolved (snapped) input.

Snapping policy is based on layer capabilities (`IsSnappable`) and active map context.

### 4) Editing actions

Edit actions (e.g. `EditLineStringAction`) must follow the same pattern as add actions:

1. derive from `InteractiveActionBase`
2. use action events (`Changed`, `Finished`, `Suspended`)
3. emit typed finish result DTO
4. use dedicated committer/orchestrator

No direct platform/UI coupling inside action logic.

## Architectural Constraints

1. Prefer shared modules (`src/PhialeGis/Core` and `src/PhialeGis/Abstractions`) for all interaction logic.
2. Platform-dependent modules (`src/PhialeGis/Platforms/*`) should only:
   - render the status bar and takeover button,
   - forward user intent (takeover request) to interaction manager,
   - avoid business logic duplication.
3. Keep naming consistent with ADR-011 (`*ActionResult`, `*Payload`, `*Service`).
4. Keep backward compatibility only where needed for compilation; no long-term fallback path to old per-target preview logic.

## Implementation Plan

### Phase 1: Contracts and session model

1. Add/extend shared contracts:
   - `ISnapService`, `SnapRequest`, `SnapResult`
   - session transfer API in `IGisInteractionManager` (e.g. `TryTakeoverInteractiveSession(...)`)
2. Refactor session registry to represent global session + input viewport semantics.
3. Keep `Suspended` and `CanBeSuspended` as first-class behavior.

### Phase 2: Manager and rendering integration

1. Refactor `GisInteractionManager` preview storage from per-target session ownership to session-level preview with viewport visibility mapping.
2. Apply snapping in one place before pointer dispatch (`PointerMove`, `PointerDown`, and ghost refresh path).
3. Add optional snap marker payload/state for overlay rendering.
4. Ensure preview invalidation updates all visible viewports for the session.

### Phase 3: Platform UX

1. Add viewport status bar UI in WPF, WinUI3, Avalonia controls.
2. Add `Przejmij rysowanie` button with visibility conditions from Decision section.
3. Wire click/tap to manager takeover API.
4. Keep map canvas gestures independent from takeover trigger.

### Phase 4: Action parity and editing foundation

1. Validate `AddLineStringAction` with new global session + snap pipeline.
2. Implement first edit action (`EditLineStringAction`) using the same contracts/pattern.
3. Reuse takeover and visibility behavior unchanged for edit action.

## Testing Strategy

### Core tests

1. Session transfer:
   - takeover changes `InputViewportId`
   - geometry state is preserved
   - suspended/resumed rules remain correct
2. Multi-viewport visibility:
   - preview visible on all eligible viewports
   - preview invalidation updates all relevant targets
3. Snapping:
   - snapping applied before action handler call
   - non-snappable layers excluded
   - no snap fallback keeps raw model position
4. Finish payload:
   - typed results remain intact and committer still persists geometry.

### Platform tests (per platform)

1. WPF, WinUI3, Avalonia:
   - status bar visibility rules
   - takeover button presence/absence rules
   - takeover click/tap path and no interference with map gestures
2. Touch-focused interaction scenarios should be explicit, not inferred from mouse-only tests.

## Consequences

### Positive

1. Multi-viewport workflows become natural for drafting and detail work.
2. Takeover is touch-friendly and explicit (low accidental activation).
3. Snapping logic becomes reusable and consistent across actions.
4. Edit actions inherit the same robust orchestration model as add actions.

### Costs / Risks

1. Session model refactor affects manager, registry, and preview invalidation paths.
2. Additional UI composition needed in each active platform.
3. Requires careful regression testing for existing add-action flows.

## Acceptance Criteria

This ADR is considered implemented when all are true:

1. Ongoing action preview is visible on multiple viewports within the same map context.
2. Action input ownership can be transferred using touch-friendly status bar button.
3. Transfer does not require keyboard and does not hijack default map click gestures.
4. Snapping is applied centrally by manager and used by `AddLineStringAction`.
5. `Suspended` and `CanBeSuspended` behavior remains operational after takeover refactor.
6. Platform tests (WPF/WinUI3/Avalonia) cover takeover UI and behavior.
7. No retained operational fallback to old per-target-only action behavior.

## Rollout Notes

1. Implement in phases, executing tests after each phase.
2. Start with safety tests before refactor.
3. Keep commits small and traceable to ADR phases.
