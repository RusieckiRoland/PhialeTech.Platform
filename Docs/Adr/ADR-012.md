# ADR: Repository Topology Must Match Physical Structure and Visual Studio Solution Topology

* **Status:** Accepted
* **Date:** 2026-03-11
* **Owners:** GIS Platform Team
* **Related:** `Docs/Adr/ADR-009.md`, `Docs/Adr/ADR-010.md`, `Docs/Adr/ADR-011.md`, `PhialeGis.Library.sln`, `PhialeTech.Components.sln`

---

## Context

The repository is currently in a hybrid state.

Two reorganizations have already started:

1. `PhialeTech` reusable components were introduced under `src/Products` and `src/Shared`.
2. Older GIS and platform projects still remain in historical root-level folders such as:
   * `PhialeGis.Library.Core`
   * `PhialeGis.Library.Geometry`
   * `Wpf`
   * `WinUI3`
   * `Avalonia`
   * `Features`

At the same time, the Visual Studio solution view is not the same as the physical repository layout.

Current failure modes:

* solution folders mix logical grouping and physical grouping,
* some projects are nested under solution folders while many remain at root level,
* `PhialeTech` is partly grouped by product while GIS projects are still grouped by history,
* platform folders are visible as the top-level architectural axis even though platform is not the primary design boundary,
* generated folders and machine-local caches (`bin`, `obj`, `GPUCache`, `DawnCache`) sit next to architectural folders and can be mistaken for part of the intended structure,
* the solution no longer communicates the real architecture of the repository.

This makes the codebase harder to reason about and increases the chance of future drift:

* a developer moves physical files without updating the solution structure,
* a developer reorganizes the solution without changing the filesystem,
* new projects are added to whichever top-level folder appears convenient,
* reusable `PhialeTech` products and GIS-specific code become mixed again.

We therefore need one explicit rule:

**the physical repository layout is the source of truth, and the Visual Studio solution layout must mirror it.**

---

## Decision

### 1. The repository will be organized by product boundary first, not by UI platform first

The primary architectural split is:

1. `PhialeGis` - GIS runtime, geometry, rendering, application-specific features
2. `PhialeTech` - reusable product components and shared component infrastructure

Platform-specific adapters are secondary and must sit under those product boundaries.

This means the repository must not be organized primarily as:

* `Wpf/`
* `WinUI3/`
* `Avalonia/`

at the top level for active code.

Platform folders remain valid, but only inside the owning product path.

### 2. The physical directory tree is the canonical architecture map

The directory tree on disk is the authoritative representation of:

* ownership,
* extraction boundaries,
* platform placement,
* product placement,
* test placement,
* demo placement.

Solution folders in Visual Studio must reflect the physical directory tree rather than define an alternative logical topology.

### 3. The top-level repository directories are fixed

The intended top-level structure is:

```text
Docs/
src/
demo/
tests/
legacy/
scripts/
tools/
```

All active source code must live under one of these roots.

The following are **not** architectural roots and must not shape the solution structure:

* `bin/`
* `obj/`
* `GPUCache/`
* `DawnCache/`
* temporary `.nuget` feeds inside source folders
* empty placeholder folders

### 4. `PhialeGis` code moves under `src/PhialeGis`

All active GIS runtime and GIS-specific UI libraries must live under:

```text
src/PhialeGis/
```

with the following internal structure:

```text
src/PhialeGis/
├─ Core/
├─ Features/
└─ Platforms/
```

#### `src/PhialeGis/Core`

Contains shared GIS runtime code and rendering infrastructure.

Projects to place here:

* `PhialeGis.Library.Abstractions`
* `PhialeGis.Library.Actions`
* `PhialeGis.Library.Core`
* `PhialeGis.Library.Domain`
* `PhialeGis.Library.Dsl`
* `PhialeGis.Library.Geometry`
* `PhialeGis.Library.Renderer.Skia`
* `PhialeGis.Library.Sync`
* `PhialeGis.WebAssets`

#### `src/PhialeGis/Features`

Contains GIS-specific feature modules that stay coupled to the GIS application and are not part of the reusable `PhialeTech` product line.

Projects to place here:

* `PhialeGis.Library.DslEditor`
* future GIS-specific features such as `DrawBox`-adjacent modules if they are not extracted as products

#### `src/PhialeGis/Platforms`

Contains GIS-specific platform adapters and GIS UI libraries.

Projects to place here:

* `PhialeGis.Library.WpfUi`
* `PhialeGis.Library.WinUi`
* `PhialeGis.Library.AvaloniaUi`

with platform nesting:

```text
src/PhialeGis/Platforms/
├─ Wpf/
├─ WinUI3/
└─ Avalonia/
```

### 5. `PhialeTech` code moves under `src/PhialeTech`

All reusable components and shared component infrastructure must live under:

```text
src/PhialeTech/
```

with the following internal structure:

```text
src/PhialeTech/
├─ Shared/
└─ Products/
```

#### `src/PhialeTech/Shared`

Contains shared component contracts, themes, and host abstractions.

Projects to place here:

* `UniversalInput.Contracts`
* `PhialeTech.ComponentHost.Abstractions`
* `PhialeTech.Styles.Wpf`

#### `src/PhialeTech/Products`

Contains reusable components grouped by product.

Projects to place here:

* `PhialeTech.Grid.*`
* `PhialeTech.ActiveLayerSelector.*`
* future `PhialeTech.LineStylePicker.*`
* future `PhialeTech.SymbolPicker.*`
* future `PhialeTech.FillStylePicker.*`

with product nesting such as:

```text
src/PhialeTech/Products/
├─ Grid/
│  ├─ Core/
│  ├─ Localization/
│  ├─ Platforms/
│  └─ MockServer/
└─ ActiveLayerSelector/
   ├─ Core/
   └─ Platforms/
```

### 6. Demo applications belong under `demo`

All demo hosts belong under `demo`, not under active product source paths.

The demo tree will be split by owning line:

```text
demo/
├─ PhialeGis/
│  ├─ Wpf/
│  ├─ WinUI3/
│  └─ Avalonia/
└─ PhialeTech/
   ├─ Shared/
   ├─ Wpf/
   ├─ WinUI3/
   └─ Avalonia/
```

This means:

* `PhialeGis.ComponentSandbox`, `PhialeGis.ComponentSandboxWinUi`, and `PhialeGis.ComponentSandbox.Avalonia` are demos and must move under `demo/PhialeGis/...`
* `PhialeTech.Components.*` stays under `demo/PhialeTech/...`

### 7. Tests belong under `tests`

All test projects must live under `tests`, grouped by owning product or system.

The intended structure is:

```text
tests/
├─ PhialeGis/
├─ PhialeTech/
└─ Performance/
```

Guiding rule:

* tests for GIS runtime live under `tests/PhialeGis/...`
* tests for reusable components live under `tests/PhialeTech/...`
* benchmark or perf-focused projects live under `tests/Performance/...` unless they are deliberately kept as separate benchmark applications under the owning product

### 8. Legacy remains physically isolated under `legacy`

The legacy platform decision from `ADR-010` remains in force.

UWP code stays under:

```text
legacy/Uwp/
```

and must not be mixed into active source or demo trees.

### 9. Solution topology must mirror the physical tree 1:1

Visual Studio solution folders must be created to match the directory tree for the included parts of the repo.

That means:

* no extra architectural folders that do not exist on disk,
* no top-level `Wpf`, `WinUI3`, or `Avalonia` folders once those projects are moved under `src/...` or `demo/...`,
* no root-level projects in the solution unless the project file is physically at repository root,
* no grouping by concept if the corresponding physical folder does not exist.

The rule is simple:

**If a project is stored in `src/PhialeTech/Products/Grid/Platforms/Wpf/...`, its solution nesting must follow `src > PhialeTech > Products > Grid > Platforms > Wpf`.**

### 10. Build output and machine-local cache folders are excluded from repository architecture

The following folders are not part of the intended repository topology and must be ignored when rebuilding solution structure:

* `bin`
* `obj`
* `GPUCache`
* `DawnCache`
* generated `.wpftmp` artifacts
* temporary local feeds stored inside source directories

If present in working trees, they must not be used as evidence for repository structure.

## Why This Structure And Not Another One

### Why not keep platform folders at repository root?

Because platform is not the primary architectural boundary.

If `Wpf`, `WinUI3`, and `Avalonia` stay at repository root, the repository suggests that:

* the product is organized by UI technology,
* each platform is equally primary,
* reusable products and GIS application code are peers to platform folders.

That is false.

The real primary split is:

* GIS runtime/application code
* reusable `PhialeTech` products

### Why not keep logical-only solution folders?

Because they drift.

A logical-only solution view is initially convenient, but over time it causes:

* folders in Visual Studio that do not exist on disk,
* projects that appear colocated even though they are far apart physically,
* new projects being added to the nearest convenient folder instead of the correct physical path,
* inaccurate mental models of extraction boundaries.

### Why not leave root-level project folders as they are?

Because the root then becomes a historical dump of projects rather than a readable repository map.

A repository root should answer a small number of questions immediately:

1. Where is the source?
2. Where are the demos?
3. Where are the tests?
4. Where is legacy code?
5. Where is documentation?

The current root does not do that clearly.

### Why split `demo` into `PhialeGis` and `PhialeTech`?

Because the repository now hosts both:

* GIS-specific sandbox hosts
* reusable component demos

Keeping those in one flat demo tree would repeat the same ambiguity that already exists in source code organization.

### Why place benchmark and mock-server assets with the Grid product?

Because they are not generic repository infrastructure.

They exist to validate, test, and demonstrate the Grid product.

That makes them product-adjacent assets, not top-level repository categories.

## Required Visual Studio Solution Topology

### Rule

For active solutions, Visual Studio solution folders must mirror the physical path from repository root.

Example:

If the project file lives at:

```text
src/PhialeTech/Products/Grid/Platforms/Wpf/PhialeTech.Grid.Wpf/PhialeTech.Grid.Wpf.csproj
```

then the solution nesting must be:

```text
src
└─ PhialeTech
   └─ Products
      └─ Grid
         └─ Platforms
            └─ Wpf
               └─ PhialeTech.Grid.Wpf
```

### Non-negotiable rules

1. No project may sit at solution root unless its `.csproj` is physically at repository root.
2. No top-level solution folder may exist if the corresponding physical top-level folder does not exist.
3. No solution folder may be used only as a conceptual tag.
4. If a physical folder is renamed or moved, the solution folder structure must be updated in the same change.
5. `Docs`, `src`, `demo`, `tests`, `legacy`, `scripts`, and `tools` are the only valid first-level solution folders for the active repository topology.

### Practical consequence

The current top-level Visual Studio folders:

* `Wpf`
* `WinUI3`
* `Avalonia`
* `Features`
* `PhialeTech`

must be replaced by physical-root mirroring after the filesystem migration.

The correct first level becomes:

* `Docs`
* `src`
* `demo`
* `tests`
* `legacy`
* `scripts`
* `tools`

## Rebuild Plan

### Phase 1: Clean non-architectural noise

1. Remove `GPUCache` and `DawnCache` from the working tree if present.
2. Ensure `bin` and `obj` remain ignored.
3. Remove empty folders such as `PhialeGis.Library.Universal` if unused.
4. Move `src/Shared/.nuget/local-feed` out of `src`.

### Phase 2: Create missing intermediate target folders

Create:

* `src/PhialeGis/Core`
* `src/PhialeGis/Features`
* `src/PhialeGis/Platforms/Wpf`
* `src/PhialeGis/Platforms/WinUI3`
* `src/PhialeGis/Platforms/Avalonia`
* `src/PhialeTech/Shared`
* `src/PhialeTech/Products/Grid`
* `src/PhialeTech/Products/ActiveLayerSelector`
* `demo/PhialeGis/Wpf`
* `demo/PhialeGis/WinUI3`
* `demo/PhialeGis/Avalonia`
* `demo/PhialeTech/Shared`
* `demo/PhialeTech/Wpf`
* `demo/PhialeTech/WinUI3`
* `demo/PhialeTech/Avalonia`
* `tests/PhialeGis`
* `tests/PhialeTech/Grid/Wpf`
* `tests/PhialeTech/Grid/MockServer`
* `tests/PhialeTech/ActiveLayerSelector/Wpf`
* `tests/Performance`

### Phase 3: Move projects physically

Move directories according to the mapping section above.

This phase must update:

* project file paths inside `.sln`
* project references inside `.csproj`
* relative paths in packaging or asset includes
* local build scripts if they reference hard-coded old paths

### Phase 4: Normalize project identity where directory names lag behind product names

After moves:

1. ensure folder names align with current product identity,
2. remove remaining `PhialeGrid` directory names where `PhialeTech.Grid` is the real product identity,
3. keep package and assembly identity consistent with `ADR-011`.

### Phase 5: Rebuild Visual Studio solution folders

For `PhialeGis.Library.sln`:

1. remove current top-level logical folders that no longer match disk,
2. create solution folders mirroring:
   * `Docs`
   * `src`
   * `demo`
   * `tests`
   * `legacy`
   * `scripts`
   * `tools`
3. nest every project under the solution folder path matching its physical directory path,
4. keep ADR files under `Docs/Adr` as solution items,
5. do not create extra logical folders.

For `PhialeTech.Components.sln`:

1. mirror only the demo-relevant physical subtree,
2. use the same physical-path rule,
3. do not reintroduce product/platform groupings that diverge from disk.

### Phase 6: Validate parity

Validation rules:

1. `Get-ChildItem` on repository roots must explain the same architecture shown by solution folders.
2. A developer can infer the physical location of a project directly from the solution nesting.
3. A developer can infer the solution nesting directly from the physical path.
4. No project is duplicated conceptually under one folder and physically under another.

## Implementation Notes For The Solution File

A `.sln` file does not mirror folders automatically.

To make the solution view match the disk layout, the rebuild must update both:

1. project paths in `Project(...) = ...` entries
2. folder nesting in `GlobalSection(NestedProjects)`

That means the migration is complete only when:

* the project has been moved on disk,
* the `.sln` project path has been updated,
* the matching solution folders exist,
* the `NestedProjects` entry points to the folder chain mirroring the path.

If only one of those steps is performed, parity is broken again.

---

## Acceptance Criteria

This ADR is considered implemented when all of the following are true:

1. The repository root contains only architectural roots and non-source tooling roots.
2. Active GIS code lives under `src/PhialeGis`.
3. Active reusable component code lives under `src/PhialeTech`.
4. Demo hosts live under `demo/PhialeGis` or `demo/PhialeTech`.
5. Test projects live under `tests/...`.
6. UWP remains under `legacy/Uwp`.
7. No active project remains at repository root.
8. Visual Studio solution folders mirror physical folders 1:1 for included projects.
9. No logical-only top-level solution folders remain.

---

## Consequences

### Positive

* The repository becomes readable without prior tribal knowledge.
* `PhialeGis` and `PhialeTech` boundaries become explicit on disk and in the solution.
* Extraction of reusable products becomes easier.
* Platform adapters stop masquerading as the primary architecture.
* The Visual Studio view becomes trustworthy.
* Future refactors become more mechanical and less interpretive.

### Trade-offs

* The migration will touch many project paths and solution references.
* Short-term churn is unavoidable.
* Any scripts or documentation that rely on old paths must be updated in the same migration wave.
* The `.sln` file will need deliberate maintenance rather than ad-hoc manual grouping.

---

## Enforcement Rule

Any future repository or solution change that introduces a mismatch between:

* physical repository structure
* Visual Studio solution structure

conflicts with this decision.

Any future project added at repository root, or any future logical-only grouping that diverges from disk, must be treated as an architectural regression.
