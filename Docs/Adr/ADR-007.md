# ADR: DSL-Driven Editor — Single Source of Truth for Validation & IntelliSense

* **Status:** Accepted
* **Date:** 2025-09-13
* **Owners:** GIS Platform Team
* **Related:** PhialeGis.Library.Dsl, PhialeGis.Library.Core, PhialeGis.Library.UwpUi.Controls

---

## Context

We are building a cross-platform DSL editor component (`PhialeDslEditor`) for UWP (later WinUI, WPF, Avalonia). The editor supports two modes:

1. **Command** — single-line input; every **Enter** submits one command.
2. **Script** — multi-line editing; **Run** executes the whole buffer or selection.

We must provide **validation (diagnostics)** and **IntelliSense (completions/hover/signature help)** with these requirements:

* No duplication of grammar knowledge in UI.
* No token lists, keywords, or parsing rules in platform projects.
* All syntax/semantics are owned by the DSL project.

---

## Decision

**The DSL project is the single source of truth for all language intelligence.** The UI/editor remains "dumb": it renders text, raises requests, and displays results. All parsing, validation, completions, hover, and execution live in **Core/DSL**, exposed via neutral interfaces.

### Interfaces (owned by Core/DSL)

```csharp
public interface IDslAnalyzer
{
    DslDiagnostics Validate(string text, int? caretOffset = null, Selection? selection = null, DslContext ctx = null);
    DslCompletions GetCompletions(string text, int caretOffset, DslContext ctx = null);
    DslHover GetHover(string text, int position, DslContext ctx = null);          // optional
    DslSignatureHelp GetSignatureHelp(string text, int position, DslContext ctx); // optional
}

public interface IDslExecutor
{
    DslResult ExecuteLine(string line, IRenderingComposition targetOrNull, DslContext ctx = null);
    DslResult ExecuteScript(string text, Selection? selection, IRenderingComposition targetOrNull, DslContext ctx = null);
}
```

**`DslInteractionManager`** (Core) orchestrates requests from editors to `IDslAnalyzer`/`IDslExecutor`, and emits **universal events** (diagnostics, completions, console output, execution state) back to editors. It also coordinates optional targeting with a viewport (`IRenderingComposition`) via `TargetDraw`.

**UI editors** (UWP/WinUI/WPF/Avalonia) expose only:

* Dependency properties (Text, Mode, IsReadOnly, TargetDraw, GisInteractionManager, …).
* Universal events (TextChanged, Command, DiagnosticsUpdated, CompletionsProvided, …).
* They never reference grammar tokens or parsing logic.

---

## Rationale

* **Maintainability:** Any DSL change updates validation and IntelliSense everywhere — no UI code changes.
* **Separation of concerns:** Platform layers host and render; Core/DSL owns language knowledge.
* **Portability:** Same DSL intelligence powers UWP, WinUI, WPF, Avalonia.
* **Testability:** Parser/analysis logic is unit-tested in isolation.

---

## Consequences

### Positive

* No token/keyword duplication in UI projects.
* Faster iteration on language features.
* Predictable behavior across platforms.
* Simpler UI code; easier to theme/replace the host.

### Negative / Trade-offs

* More UI ⇄ Core round-trips for live features (debounce required).
* Analyzer must support incremental/fast validation for good UX.
* Requires stable, versioned DTOs for diagnostics/completions.

---

## How It Works (high level)

### Command Mode

* **Enter** → UI raises `CommandUniversal(line, targetOrNull)` → `DslInteractionManager.ExecuteLine(...)`.
* If diagnostics contain **Error** → UI shows red border & tooltip (+ Problems panel if enabled).
* Else → effects applied (single target or broadcast), Output appended.

### Script Mode

* **Run** (button or Ctrl+Enter) → `ExecuteScript(text | selection, targetOrNull)`.
* Live lint: debounced `TextChanged` → `Validate(text)` → `DiagnosticsUpdatedUniversal`.

### IntelliSense (both modes)

* Debounced caret/typing → `GetCompletions(text, caret, ctx)` from DSL.
* UI shows a popup list; **Tab/Enter** inserts; **Esc** closes.
* Hover/signature help requested similarly (optional).

---

## API/DTO Rules

* **Diagnostics:** DSL returns line/column/span + message + severity; `DslInteractionManager` maps to universal diagnostics and raises `DiagnosticsUpdatedUniversal`.
* **Completions:** DSL returns items `{ label, insertText, detail, kind, replaceSpan }`; UI only renders.
* **Context:** `DslContext` may include runtime info (available layers/fields) so suggestions are semantic; UI does not derive this.

---

## Alternatives Considered

1. **UI-side keyword lists / regex validation**
   ❌ Drifts from grammar; duplicative; brittle; multiplies platform code.

2. **Split logic: syntax in UI, semantics in Core**
   ❌ Two sources of truth; complex failure modes; higher maintenance.

3. **Language Server Protocol (LSP)**
   ✅ Clean separation, but heavier infra; overkill for now. Revisit later if needed.

---

## Risks & Mitigations

* **Latency in live features** → debounce (120–250 ms), incremental parse caches in DSL.
* **Version drift** between UI and DSL → version/tag interfaces; add `IDslAnalyzer.Version`.
* **Many editors registered** → `DslInteractionManager` manages the active editor and per-editor state; single-active execution if required.

---

## Work Plan (phased)

1. **MVP Command** — wire `ExecuteLine(...)`; show red border & tooltip on errors; append output.
2. **Live Validation** — debounced `Validate(text)`; render `DiagnosticsUpdatedUniversal` (squiggles later).
3. **Completions** — wire `GetCompletions(text, caret)`; popup & accept; no token lists in UI.
4. **Script Mode** — run whole/selection; execution state events; output panel.
5. **Hover/Signature Help** (optional).
6. **Panels** — Problems & Output; theming.

---

## Affected Components

* **PhialeGis.Library.Dsl** (ANTLR grammar, analyzer, executor)
* **PhialeGis.Library.Core**

  * `DslInteractionManager` (new)
  * `GisInteractionManager` (composition with DSL manager)
* **UI controls** per platform (UWP now; later WinUI/WPF/Avalonia) — only hosting & event wiring.

---

## Testing Strategy

* **Unit tests** in DSL: parse success/fail, diagnostic spans, completion sets by caret.
* **Integration tests** for `DslInteractionManager`: editor → analyzer → events.
* **UI smoke tests**: Enter submits; diagnostics toggle border; completions popup lifecycle.

---

## Open Questions

* Do we need **cancel/interrupt** for long-running script execution?
* Where to persist **history** (Command) — DSL manager or app layer?
* Consider an **LSP boundary** in the future if we outgrow in-proc.

---

## Glossary

* **DSL** — Domain-specific language for GIS operations.
* **Universal Events** — Platform-agnostic events/DTOs used by UI and Core.
* **TargetDraw** — Optional viewport adapter (`IRenderingComposition`) selected by the editor.
* **Problems/Output Panels** — Optional UI panels for diagnostics and execution logs.
