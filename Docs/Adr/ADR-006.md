# ADR-006: Case-Insensitive DSL Handling

## Status
Accepted (2025-09-05), synchronized with current implementation (2026-02-16)

## Context
The PhialeGIS DSL targets non-programmer users. Requiring exact letter casing for commands is unfriendly and causes avoidable errors.

Current implemented commands include:
- `ZOOMIN`
- `ZOOMOUT`
- `ZOOM <number>`
- `ADD LINESTRING`

Users should be able to type mixed casing (for example: `zoomin`, `ZoomOut`, `add linestring`).

## Decision
DSL keywords are treated as case-insensitive primarily through lexer-level support in ANTLR.

1. **Lexer-level case-insensitivity**
- `PhialeDslLexer.g4` uses `options { caseInsensitive = true; }`.
- This is the current and intended primary mechanism for case-insensitive command handling.

2. **Temporary runtime preprocessing**
- `DslEngine.ParseSingle(...)` still contains `UpperOutsideQuotes(...)`.
- This method is now treated as transitional implementation code, not as the target architectural mechanism.
- It is planned for removal after parser/runtime tests confirm that lexer-level case-insensitivity fully covers the required command paths.

3. **Tests**
- Tests should validate the actual runtime/parser behavior based on ANTLR case-insensitive lexing.
- Any remaining preprocessing in tests should be treated as temporary and aligned with the runtime cleanup plan.

## Rationale
- Better UX for command entry.
- Grammar remains compact and readable.
- ANTLR provides built-in case-insensitive lexer support, so custom input normalization should not remain the long-term basis of this behavior unless proven necessary.

## Implementation Details
- Lexer option:
  - `PhialeGis.Library.Dsl/Grammar/PhialeDslLexer.g4`
- Transitional runtime preprocessing:
  - `PhialeGis.Library.Dsl/Api/DslEngine.cs`
- Transitional test preprocessing:
  - `PhialeGis.Library.Tests/Dsl/Visitor/DslVisitorTests.cs`

## Consequences
### Positive
- Mixed-case commands work in practice.
- The primary mechanism is explicit and based on ANTLR grammar support.
- The architecture is moving toward a simpler and clearer responsibility model.

### Negative / Risks
- Transitional duplication still exists until preprocessing is removed.
- Until cleanup is completed, ownership of case-insensitive behavior is partially split between grammar and legacy preprocessing.

## Follow-Up
- Verify through parser/runtime tests that `options { caseInsensitive = true; }` is sufficient for all supported command paths.
- Remove `UpperOutsideQuotes(...)` from production code after tests confirm it is no longer needed.
- Remove corresponding temporary preprocessing from tests.
- Revalidate diagnostics and completion behavior after cleanup.

## Alternatives Considered
1. Runtime preprocessing only (without lexer option).
2. Lexer-only case-insensitivity (target direction).
3. Verbose grammar-level fragment approach (`[Aa]`, `[Bb]`, ...), rejected for readability and maintenance reasons.

## Testing
- Parsing tests should validate mixed-case commands directly.
- Visitor tests should validate expected DTO mapping for command forms and casing variants.
- Cleanup of transitional preprocessing must be covered by regression tests before removal.

## Notes
- This ADR reflects the current implementation direction: ANTLR lexer-level case-insensitivity is the intended solution, while `UpperOutsideQuotes(...)` remains temporary code scheduled for removal after test verification.
